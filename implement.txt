# CRITICAL BUG TO FIX FIRST: Zeitwerk Naming Conflict
################################################################################

PROBLEM:
Two files define the constant `Current`:
  1. app/controllers/concerns/current.rb   → `module Current` (ActiveSupport::Concern)
  2. app/models/current.rb                 → `class Current < ActiveSupport::CurrentAttributes`

At runtime, Zeitwerk loads the concern FIRST. `Current` is a Module, NOT a Class.
The ActiveSupport::CurrentAttributes class is DEAD CODE — never loaded.

Proof:
  `bundle exec rails runner "puts Current.class"` → Module
  `bundle exec rails runner "puts Current.respond_to?(:preference)"` → false

15+ application controllers do `include ::Current`, so the module cannot be renamed.

SOLUTION:

1. DELETE `app/models/current.rb` — it is dead code. Its contents (actor, actor_type,
   session, token, etc.) were never functional. Step 9 will implement these properly.

2. MODIFY `app/controllers/concerns/current.rb` — Add `preference` accessor with
   Thread.current storage (same pattern ActiveSupport::CurrentAttributes uses internally):

```ruby
# typed: false
# frozen_string_literal: true

module Current
  extend ActiveSupport::Concern

  class << self
    def preference
      Thread.current[:_current_preference] ||= Current::Preference::NULL
    end

    def preference=(value)
      Thread.current[:_current_preference] = value
    end

    def reset
      Thread.current[:_current_preference] = Current::Preference::NULL
    end
  end

  included do
    after_action :_reset_current_state
  end

  private

  def set_current
  end

  def _reset_current_state
    Current.reset
  end
end
```

3. KEEP `app/models/current/preference.rb` AS-IS — it already uses `module Current`.

4. KEEP `test/unit/current/preference_test.rb` AS-IS — it calls `Current.reset` and
   `Current.preference` which will now resolve to the concern's class methods.

VERIFICATION:
  bundle exec rails test test/unit/current/preference_test.rb  → 10 pass
  bundle exec rails runner "puts Current.preference.null?"     → true
  bundle exec rails runner "puts Current.preference.language"  → ja

# STEP 4: Preference JWT/Token Implementation
################################################################################

CONTEXT:
The app has TWO separate JWT systems:
  A) Auth JWT (app/services/auth/token_claims.rb) — for authentication
     - Claims: sub, act, typ, iss, aud, sid, jti, iat, nbf, exp
     - NO preference data currently
  B) Preference JWT (app/controllers/concerns/preference/base.rb) — for preferences
     - Claims: preferences{lx,ri,tz,ct}, public_id, preference_type, host, jti, etc.
     - Separate cookies: __Secure-preference_access, __Secure-preference_refresh

The goal is to ADD a `prf` claim to the Auth JWT (system A) so that
`Current.preference` can be populated from the auth token without needing the
separate preference JWT. The preference JWT system (B) continues to exist for
guests/unauthenticated users.

## Item 1: Add `prf` claim to Auth::TokenClaims.build

FILE: app/services/auth/token_claims.rb

Add a `prf` claim containing a preference snapshot. This claim should be a hash
with the short-code keys used throughout the system:

```ruby
def build(resource:, session_public_id:, resource_type:, issued_at:, access_token_ttl:,
          expires_at: nil, preference: nil)
  # ... existing payload construction ...
  if preference
    payload["prf"] = {
      "lx" => preference[:language] || preference["lx"],
      "ri" => preference[:region] || preference["ri"],
      "tz" => preference[:timezone] || preference["tz"],
      "ct" => preference[:theme] || preference["ct"],
    }.compact
  end
  payload
end
```

Also add an extraction method:
```ruby
def preference(payload)
  payload&.dig("prf")
end
```

The `preference:` parameter is optional — existing callers continue to work.

WHERE CALLED: In auth/base.rb, the `build_access_token` or equivalent method calls
`Auth::TokenClaims.build`. That call site needs to pass `preference:` when available.
Look for where `Auth::TokenClaims.build` is called in auth/base.rb and add:

```ruby
preference: build_preference_snapshot(resource)
```

The `build_preference_snapshot` helper should load UserPreference/StaffPreference
for the resource and return a hash like:
  { language: "ja", region: "jp", timezone: "Asia/Tokyo", theme: "sy" }

If no preference exists, return nil (the prf claim will be omitted).

## Item 2: Create Current::Preference.from_jwt(prf_claim)

FILE: app/models/current/preference.rb

Add a class method that constructs a Preference from the JWT prf claim:

```ruby
def self.from_jwt(prf_claim, cookie: NULL_COOKIE)
  return NULL unless prf_claim.is_a?(Hash)

  new(
    language: prf_claim["lx"] || DEFAULTS[:language],
    region:   prf_claim["ri"] || DEFAULTS[:region],
    timezone: prf_claim["tz"] || DEFAULTS[:timezone],
    theme:    prf_claim["ct"] || DEFAULTS[:theme],
    cookie:   cookie,
  )
end
```

## Item 3: Wire Current.preference into request lifecycle

In `app/controllers/concerns/auth/base.rb`, after the current resource is resolved
from the JWT token (in `load_current_resource` or `load_from_token`), populate
`Current.preference` from the JWT `prf` claim:

```ruby
# After successful JWT decode and resource load:
prf_claim = Auth::TokenClaims.preference(payload)
if prf_claim
  Current.preference = Current::Preference.from_jwt(prf_claim)
end
```

For guests (no auth token), `Current.preference` stays as `Current::Preference::NULL`
which is set by the concern's reset mechanism.

## Item 4: Implement reissue_access_token!

In `app/controllers/concerns/auth/base.rb`, add a method that reissues the access
token with updated preference data. This is called when preferences change (theme
toggle, language switch, etc.):

```ruby
def reissue_access_token!(preference_data = nil)
  return unless @current_resource

  token_record = current_token_record
  return unless token_record

  preference_snapshot = preference_data || build_preference_snapshot(@current_resource)

  access_token = generate_access_token(
    resource: @current_resource,
    token_record: token_record,
    preference: preference_snapshot,
  )

  set_access_cookie(access_token, ACCESS_TOKEN_TTL.from_now)

  # Update Current.preference immediately
  Current.preference = Current::Preference.from_jwt(
    preference_snapshot.transform_keys { |k| { language: "lx", region: "ri", timezone: "tz", theme: "ct" }[k] || k.to_s }
  ) if preference_snapshot
end
```

NOTE: Look at how access tokens are currently generated in auth/base.rb's login and
refresh flows. Reuse the same JWT signing infrastructure. The method names above are
illustrative — use whatever the existing codebase calls.

# STEP 5: Preference Model Consolidation
################################################################################

CONTEXT:
Currently there are 5 tiers of preference models with child records:
  - AppPreference (preference DB) — session-based for app users
  - OrgPreference (preference DB) — session-based for staff
  - ComPreference (preference DB) — session-based for corporate/public
  - UserPreference (principal DB) — 1:1 per user
  - StaffPreference (principal DB) — 1:1 per staff

Each has 4 child records (timezone, language, region, colortheme) plus cookie consent.
That's ~73 model files. The "Adoption" concern syncs between them at login time.

GOAL: UserPreference and StaffPreference (principal DB, 1:1) become the source of truth
for logged-in users. AppPreference/OrgPreference/ComPreference remain for guest sessions
but are simplified.

TASKS:

1. Ensure UserPreference/StaffPreference have all needed columns:
   - language, region, timezone, theme (direct columns, not child records)
   - consented, functional, performant, targetable, consent_version, consented_at
   - Check existing schema in db/principal_schema.rb

   If child-record pattern is currently used (user_preference_languages, etc.), consider
   whether to keep it or flatten to direct columns. Direct columns are simpler.

2. Update login-time sync (`preference/adoption.rb`):
   - On login: copy from AppPreference/OrgPreference child records → UserPreference/StaffPreference
   - On subsequent requests: read from UserPreference/StaffPreference (via JWT prf claim)
   - Bidirectional sync based on updated_at (newer wins)

3. The `build_preference_snapshot` helper (from Step 4) should read from
   UserPreference/StaffPreference, falling back to AppPreference if needed.

4. DO NOT delete AppPreference/OrgPreference/ComPreference yet — they still serve guests.

5. Update tests that reference the old multi-model sync patterns.

IMPORTANT: This is a large refactor. Be conservative. If the existing adoption.rb
sync works, keep it working. The main goal is ensuring the JWT prf claim is populated
from the right source at login time.

# REFERENCE: Preference Short Codes
################################################################################

lx = language: "ja" (Japanese), "en" (English), etc.
ri = region: "jp" (Japan), "us" (United States), etc.
tz = timezone: "Asia/Tokyo", "America/New_York", etc.
ct = colortheme: "sy" (system), "dr" (dark), "li" (light)

  - Cookie values (ct cookie)
  - Auth JWT prf claim (Step 4 adds this)
